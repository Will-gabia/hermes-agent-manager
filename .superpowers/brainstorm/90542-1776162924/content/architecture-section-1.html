Section 1 · System architecture

Recommended v1 shape: one Hono control-plane service, synchronous orchestration, template-only provisioning, and full Caddyfile generation from database state.

Centralized Remote Docker Manager
Access Layer

Admin UI

Login, config, server, and container management screens.

Admin Auth

Single admin credential from .env backed by session-based access.

Central Manager · Hono + TypeScript

Orchestration Service

Runs Create/Delete/Start/Stop/Status flows in request time and records final state.

Prisma + SQLite

Config, servers, templates, containers, and operation history.

Cloudflare DNS

Creates and deletes domain records under the configured base domain.

Caddyfile Generator

Always regenerates the full routing file from DB state; v1 stops at file generation.

Remote Execution Layer

docker-container-api Nodes

Each registered server exposes a remote Docker control API with predefined container templates.

Managed Containers

Each container is assigned a generated slug, domain, API token, service port, and server binding.

Request flow
1. Admin action

Create/Delete from UI or API.

2. Remote container step

Call docker-container-api on the selected server.

3. DNS step

Apply Cloudflare record for the generated domain.

4. Routing step

Regenerate the full Caddyfile from active containers.

5. Persist result

Save success or error state for inspection.

Why this fits v1

  • Simple operational model
  • Matches synchronous admin expectations
  • Keeps Caddy deployment concerns outside the app

Known constraints

  • Longer request times during create/delete
  • Needs strong partial-failure tracking
  • Template-only provisioning by design
